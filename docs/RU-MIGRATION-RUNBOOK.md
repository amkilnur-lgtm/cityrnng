# Перенос в ру-сегмент (доступность без VPN + 152-ФЗ)

Дата: 2026-07-10. Статус: быстрый фикс доступен сразу; полный переезд — по шагам ниже.

## Диагноз

- `cityrunning.online` и `staging.cityrunning.online` проксируются **Cloudflare**
  (A-записи 104.21.x / 172.67.x, NS `*.ns.cloudflare.com`). Российские операторы
  массово замедляют/режут диапазоны Cloudflare → сайт из РФ без VPN не открывается.
- Origin (datacheap NL, `103.112.68.202`) при этом **отвечает напрямую с валидным
  Let's Encrypt-сертификатом** — TLS терминирует Caddy на хосте, не Cloudflare.
  Проверено: `curl --resolve staging.cityrunning.online:443:103.112.68.202 …` → 200.

## Часть 1 — быстрый фикс (2 минуты, без переезда)

В дашборде Cloudflare → DNS → записи `staging` и `@` (прод):
**переключить оранжевое облако на серое** («DNS only»). Прокси выключается,
трафик идёт напрямую на origin, HTTPS продолжает работать (Caddy + LE).

Проверка: с телефона на мобильном интернете, **без VPN**, открыть
`https://staging.cityrunning.online`. Открылось — бегуны и сканер работают
уже сегодня.

Риски: если конкретный оператор режет и прямой NL-диапазон — поможет только
Часть 2. И это не решает 152-ФЗ (персональные данные граждан РФ должны
первично храниться в РФ) — NL-база остаётся юридическим долгом.

## Часть 2 — переезд на российский хостинг

### Провайдер

- **Yandex Cloud** — канон из CLAUDE.md (Compute Instance + опционально
  Managed PostgreSQL). Дороже, зато managed-БД и рост.
- **Timeweb Cloud / Selectel** — дешевле (~700–1200 ₽/мес за 2 vCPU / 4 GB /
  60 GB NVMe), для текущего масштаба (десятки юзеров) более чем.

Минимальные требования VM: Ubuntu 22.04+, 2 vCPU, 4 GB RAM, 40+ GB SSD,
публичный IPv4, порты 22/80/443.

### Шаги

1. **Провижининг**: создать VM, завести юзера `cityrnng` (sudo, ssh-ключ —
   можно переиспользовать `~/.cityrnng-deploy/staging_deploy_key.pub`),
   отключить парольный SSH, ufw: allow 22/80/443.
2. **Софт**: docker + docker compose plugin, caddy (пакет дистрибутива).
   Caddyfile — копия текущего `/etc/caddy/Caddyfile` (тот же virtual host).
3. **Перенос данных** (окно ~5 минут даунтайма):
   ```bash
   # на старом хосте
   docker exec cityrnng-postgres pg_dump -U cityrnng -Fc cityrnng_staging > backup.dump
   scp backup.dump новый-хост:
   # на новом: поднять compose (postgres+redis), затем
   docker exec -i cityrnng-postgres pg_restore -U cityrnng -d cityrnng_staging --clean < backup.dump
   ```
4. **Секреты**: скопировать `.env` со старого хоста (`APP_ENV_FILE` quirk из
   DEPLOY-RUNBOOK); осиротевшие `STRAVA_*` при переносе выбросить.
5. **CI/CD**: в GitHub Secrets поменять `STAGING_HOST` на новый IP (и/или
   завести `PROD_*`-набор для прод-хоста — прод деплоится вручную по правилам).
   `ssh-keyscan` нового хоста произойдёт в workflow автоматически.
   Замечание: пулы `ghcr.io` из РФ бывают медленными/нестабильными — если
   деплой начнёт падать по таймаутам, варианты: build на самом хосте
   (`docker compose build`) или зеркало в YC Container Registry.
6. **DNS cutover**: в Cloudflare сменить A-записи на новый IP, режим —
   **DNS only** (серое облако, см. Часть 1). TTL заранее снизить до 300.
   Caddy на новом хосте сам выпустит LE-сертификат при первом заходе
   (порт 80 должен быть открыт до переключения DNS — HTTP-01).
7. **Проверки после переезда**:
   - `GET /api/v1/health` → 200;
   - вход по паролю + `/me` отдаёт `checkinCode`;
   - `POST /integrations/checkin/scan` с ключом сканера → ожидаемый результат;
   - `_prisma_migrations` — все применены;
   - с телефона без VPN из РФ всё открывается.
8. **Старый хост**: неделю держать как горячий бэкап, затем погасить.

### Сканер (малина)

Ничего менять не нужно: он ходит по доменному имени. После смены DNS
автоматически пойдёт на новый хост; накопленный офлайн-буфер дошлётся сам
(зачёт считается по времени скана, не доставки).
